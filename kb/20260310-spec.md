# desktopctl – Project Specification

## Overview

**desktopctl** is a FOSS command-line tool that allows agents and developers to control desktop applications through a deterministic CLI interface.

The goal is to provide a universal control layer for desktop software, similar to how:

* `kubectl` controls Kubernetes
* `psql` controls Postgres
* `docker` controls containers
* `playwright` controls browsers

desktopctl exposes a consistent interface that allows agents to:

* open applications
* observe UI state
* interact with UI elements
* read information from the screen
* type text and trigger actions

The system uses a **hybrid automation model**:

1. Structured OS interfaces (accessibility APIs) when available
2. Computer vision when structured interfaces are unavailable

Over time the system builds a **shared knowledge graph of UI elements and interaction patterns**, improving recognition and automation reliability across applications and versions.

---

# Core Design Principles

## Agent-first

desktopctl is designed primarily for **automation agents and scripts**, not human operators.

Commands must be:

* deterministic
* composable
* machine readable
* idempotent where possible

All commands must support **structured JSON output**.

---

## Universal Application Support

desktopctl must be able to interact with:

* native desktop applications
* Electron apps
* browsers
* IDEs
* productivity software
* legacy software
* applications without automation APIs
* games (future)

The system must operate through **screen observation and input simulation**, ensuring compatibility with any software.

---

## Hybrid Automation

desktopctl uses multiple observation methods:

| Method             | Purpose                    |
| ------------------ | -------------------------- |
| Accessibility APIs | structured UI tree         |
| Computer vision    | detect visual UI elements  |
| OCR                | read visible text          |
| Layout analysis    | infer UI structure         |
| Embeddings         | identify known UI elements |

Automation always prefers **structured UI access** when available and falls back to **vision-based detection** otherwise.

---

## Collaborative Skill Learning

desktopctl supports a distributed knowledge system where the community contributes:

* UI element embeddings
* interaction patterns
* automation flows
* layout models
* application-specific heuristics

This creates a **shared interaction graph** allowing the system to improve recognition and automation reliability.

The system may optionally synchronize:

* UI element embeddings
* action success patterns
* application layouts

Users can opt in to sharing anonymized embeddings.

# CLI Interface

The CLI is the primary interface for automation.

All commands follow the structure:

```
desktopctl <resource> <action> [arguments]
```

Example:

```
desktopctl app open Messages
desktopctl ui click --text "Send"
desktopctl screen read --region chat
```

Commands must support:

```
--json
```

to return structured machine-readable output.

# Command Categories

## Application Control

Control lifecycle and focus of applications.

### Open Application

```
desktopctl app open <application>
```

Example:

```
desktopctl app open Messages
desktopctl app open "Visual Studio Code"
```

---

### List Running Applications

```
desktopctl app list
```

Output:

```
[
  { "name": "Messages", "pid": 4412 },
  { "name": "Safari", "pid": 3291 }
]
```

---

### Focus Application

```
desktopctl app focus <application>
```

---

### Close Application

```
desktopctl app close <application>
```

# UI Interaction

Interact with UI elements inside applications.

---

## Click Element

Click a UI element detected by:

* text
* label
* icon
* embedding
* location

```
desktopctl ui click --text "Send"
```

```
desktopctl ui click --label "Family Chat"
```

```
desktopctl ui click --embedding send_button
```

---

## Type Text

```
desktopctl type "Hello world"
```

---

## Press Key

```
desktopctl key press Enter
desktopctl key press Cmd+Shift+P
```

---

## Hover Element

```
desktopctl ui hover --text "File"
```

# Screen Observation

Commands that read information from the UI.

---

## Screenshot

```
desktopctl screen capture
```

Optional region:

```
desktopctl screen capture --region chat_window
```

---

## Read Text

Extract visible text using OCR or accessibility APIs.

```
desktopctl screen read
```

Example:

```
desktopctl screen read --region chat
```

Output:

```
{
  "text": "Dinner at 7?"
}
```

---

## Query UI Elements

Return detected UI elements.

```
desktopctl ui list
```

Output example:

```
[
  { "type": "button", "text": "Send" },
  { "type": "input", "placeholder": "Message" }
]
```

# High-Level Actions

Composite commands built from lower-level primitives.

Example:

```
desktopctl message send \
  --app Messages \
  --chat "Family" \
  --text "This is funny"
```

This internally performs:

1. open app
2. focus chat
3. type message
4. press send

---

# Skills System

Skills represent reusable automation knowledge.

Example:

```
desktopctl skill run messages.send
```

Skill definition example:

```
messages.send:
  open app Messages
  open chat {{chat}}
  type {{message}}
  click Send
```

Skills may be distributed and shared across users.

desktopctl maintains embeddings representing UI elements.

Embeddings may represent:

* buttons
* icons
* menu entries

* widgets

Embeddings allow recognition even when:

* UI moves
* theme changes
* resolution changes

# Interaction Graph

desktopctl maintains a knowledge graph describing:

* UI element relationships
* interaction sequences
* layout patterns
* state transitions

Example relationships:

```
Messages app
   ├─ chat list
   ├─ message view
   ├─ input box
   └─ send button
```

This graph improves reliability over time.

# Local vs Cloud Processing

desktopctl supports both:

## Local Mode

All recognition and automation runs locally.

Uses local models for:

* UI detection
* OCR
* embeddings

No data leaves the machine.

---

## Cloud Mode

Users may optionally configure external inference.

Example:

```
desktopctl config set cv_provider openrouter
```

This allows using remote models for:

* computer vision
* embedding generation
* UI analysis

* permission prompts
